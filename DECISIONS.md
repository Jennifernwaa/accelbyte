# Webhook Delivery Service Decisions

This service uses a pragmatic at-least-once delivery model. Webhooks are asynchronous and delivered to third-party endpoints outside our control, so the service guarantees that an event is persisted and retried until it is either delivered or reaches a terminal failure state. We do not claim exactly-once semantics end-to-end. Instead, we rely on a durable event ID and receiver-side idempotency to make duplicates safe when the endpoint is retried after a transient outage.

Retry behavior is exponential backoff with jitter, capped at a bounded retry budget. A delivery is considered terminal when the endpoint returns a permanent client error (for example 4xx such as 400, 404, or 410) or when the retry budget is exhausted. We keep the event record for inspection and replay rather than dropping it silently.

For long outages, the current implementation keeps the event record durable in SQLite and retries it independently of the incoming request path. That is a good baseline for reliability, but it is not a full isolation strategy. In the present single-process version, the worker loop reads from a global queue of pending or retrying events. That means a customer with many failing events can still dominate the worker’s attention, especially when the queue is globally ordered by next retry time. A stronger production design would use per-customer or per-endpoint partitions and a fair round-robin scheduler so that one unhealthy customer does not starve other customers.

Ordering is intentionally not globally guaranteed across all customers. The current service preserves a simple, predictable single-process workflow, but it does not provide a strict ordering contract across the entire system. If order matters for a customer’s own event stream, the next step would be to enforce a per-customer queue or a per-customer sequence number rather than a single global queue.

The current implementation is intentionally small and interview-friendly, but it has a few known limitations that would be addressed in production. The worker is a single thread, so one slow HTTP timeout can temporarily stall background processing. There is no structured logging today, which makes troubleshooting harder when a delivery fails repeatedly. The retry parameters are still hardcoded, and the code does not use a separate processing lease status (for example `processing`) before the outbound HTTP request, which means a crash between the send and the SQL update could create a duplicate delivery. These are deliberate trade-offs for a 3-hour implementation, and the next production revision would add per-customer queue partitioning, concurrency, structured logging, configuration-driven retry settings, and a lease-based delivery state machine.
